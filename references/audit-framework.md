# Xiaobai Backend Acceptance Framework

Use this table as the required review skeleton. Fill every row with evidence and mark unknowns as `Unverified`.

| # | Direction | Specific checks | Check method | Acceptance criteria | Passing criteria | Output content |
|---|---|---|---|---|---|---|
| 1 | Business flow coverage | Core user journeys, success path, failure path, front-end actions mapped to backend actions | Map `user action -> API -> data change -> response`; inspect routes and product docs | Each core journey has a backend path and clear result | All MVP journeys can be explained and verified at L2+; launch-critical flows at L4+ when runnable | Journey map, missing steps, blocked flows, evidence level |
| 2 | Domain model and business rules | Core entities, ownership, lifecycle, invariants, business constraints | Inspect models, schemas, service logic, docs; compare to product goal | Business concepts are explicit and not mixed together | Each core entity has clear fields, relations, and rules; no major ambiguity | Entity list, rule list, confused concepts, fix suggestions |
| 3 | State machine, transaction, concurrency, and idempotency | State transitions, duplicate submit, atomic operations, race conditions | Inspect update paths, DB transactions, unique keys, locking, idempotency keys | Operations that change money, quota, ownership, inventory, or generated content are safe under retries and concurrent use | Illegal transitions blocked; repeated requests do not duplicate side effects; critical multi-step writes are transactional | State table, unsafe transitions, idempotency gaps, transaction risks |
| 4 | Database design | Tables, columns, indexes, relations, migrations, timestamps, soft delete needs | Inspect schema, migrations, ORM models, seed/init scripts | Database supports current business and basic growth | Required entities exist; ownership fields exist; indexes cover key queries; migrations are reproducible | Table summary, missing fields, relation risks, migration status |
| 5 | API design | Routes, methods, parameters, response body, pagination, module coverage | Inspect route definitions, API docs, OpenAPI, controllers | Front end and external clients can complete required actions | APIs cover core actions; no obvious duplicate/confusing routes; parameters and responses are usable | API inventory, missing APIs, confusing APIs, examples |
| 6 | API contract and version stability | Unified response shape, error shape, naming, compatibility, versioning | Compare route responses and error handlers; inspect shared types/docs | Front end can rely on stable contracts | Consistent response envelope and error format; breaking changes are controlled or documented | Contract issues, inconsistent fields, versioning need |
| 7 | Feature module boundaries | Auth, user, business modules, payment, AI tasks, files, admin, shared utilities | Inspect folder structure, imports, service boundaries | Code modules match business responsibilities | No critical business logic hidden in random files; shared code is not tangled | Module map, mixed responsibilities, maintainability risks |
| 8 | Permission and security | Authentication, authorization, secret handling, input validation, rate limit, CORS | Inspect middleware, env config, validators, dependency security, auth tests | Protected resources cannot be accessed or changed by unauthorized users | Login-required APIs are protected; roles are enforced; secrets are not committed; sensitive input is validated | Security findings, affected routes, P0/P1 risks |
| 9 | Data ownership and isolation | User/team/company ownership, tenant boundaries, admin scope, query filters | Inspect every query/update/delete path for ownership filters | Users only access data they own or are allowed to manage | All sensitive reads/writes filter by owner/tenant; admin power is explicit and limited | Isolation matrix, risky queries, possible data leaks |
| 10 | Error handling | Missing params, invalid state, not found, permission denied, external failure, DB failure | Inspect exception handling and response formats; run tests or API calls when possible | Failures return understandable, safe, consistent errors | No raw stack traces to users; errors are logged; client receives actionable status and message | Error matrix, inconsistent handling, user-facing impact |
| 11 | External dependencies, degradation, and compensation | AI APIs, payment, email/SMS, storage, OAuth, webhooks, retries, timeout | Inspect clients, timeout config, retry logic, callbacks, compensation records | Third-party failure does not silently corrupt core data | Timeouts exist; retries are safe; callbacks are verified; manual recovery path exists for critical failures | Dependency list, failure modes, fallback/compensation gaps |
| 12 | Logs, monitoring, and observability | Request ID, user ID, route, latency, error stack, business events, health check | Inspect logging, middleware, health endpoints, deployment logs | Operators can locate failures after users report them | Key failures log enough context without leaking secrets; health check exists | Observability checklist, missing logs, diagnosis gaps |
| 13 | Deployment, migration, backup, and recovery | Env vars, startup command, DB migration, rollback, backup, seed data, health check | Inspect README, Docker/Render/Vercel/Fly files, CI/CD, migration scripts | A new environment can be started and recovered predictably | Required env vars documented; migrations run; backup/recovery plan exists for important data | Deployment readiness, missing env, recovery risks |
| 14 | Performance, capacity, async jobs, and test evidence | Pagination, indexes, slow queries, large files, AI generation, background jobs, test coverage | Inspect query patterns, task queues, tests; run available test suite when safe | MVP can handle expected usage without blocking core requests | Slow/long tasks are async or bounded; list APIs paginate; tests cover core flows; verification evidence is stated | Performance risks, async needs, test proof, evidence summary |

## Pass/Fail Guidance

- A launch-critical item with only L0 or L1 evidence cannot be marked passed.
- Any P0 means overall verdict is `Do not launch`.
- Any unresolved P1 means overall verdict is at most `Pass with fixes`.
- If code cannot be inspected, use `Cannot verify` unless the user only requested a planning review.
- For non-technical users, translate each major issue into: `what could go wrong`, `who is affected`, and `what to ask the AI developer to fix`.
